# Per-User VM integration authentication and state contracts

Research for [#61](https://github.com/aviralmansingka/dotfiles/issues/61), under the isolation decisions in map [#47](https://github.com/aviralmansingka/dotfiles/issues/47).

## Result

Every credential-bearing integration state must have exactly one owner inside its User VM; the Golden Image contains none of it. Google Workspace can authorize independently in each VM. WhatsApp's credential belongs to the transport bridge, not its MCP server or Pi transport daemon. The configured Granola MCP is only a reader of Granola desktop-app state and currently cannot satisfy a Linux User VM contract or read the encrypted state observed on this Mac.

The checked-in `pi/.pi/agent/mcp.json` also launches `workspace-mcp` and `granola-mcp-plus` without version pins. Paths and behavior below describe Workspace MCP v1.22.1, `granola-mcp-plus` v1.2.0, and the local WhatsApp checkout at upstream commit `7d6a06d` plus uncommitted local bridge changes. Pin versions before treating these as stable deployment contracts.

## State boundary: WhatsApp MCP versus transport

The Go bridge is both the WhatsApp linked-device transport and the local data/API provider. It alone owns `whatsapp.db`. The Python WhatsApp MCP is a stateless stdio facade over `messages.db` plus the bridge's localhost REST API. The checked `pi-whatsapp-daemon.py` is a second consumer that polls `messages.db`, sends commands to Pi, and stores only `last_rowid` in `~/.pi/agent/pi-whatsapp-state.json`. Back up, restore, revoke, and verify these layers separately:

1. **Credential:** `whatsapp.db` — linked-device keys/session; bridge-owned secret.
2. **Transport data:** `messages.db` and downloaded media — private message content, not credential.
3. **Pi transport state/config:** daemon checkpoint and allowlist env — routing state, not credential.
4. **MCP:** no independent WhatsApp credential or persistent session.

## Documented facts, observations, and unresolved safe experiments

### Documented/source facts

- Workspace MCP stores local credentials with mode `0600`, requests offline access, validates OAuth state/PKCE, refreshes and rewrites credentials, and fetches Google userinfo during callback.
- The WhatsApp bridge uses whatsmeow's SQL device store. Presence of a stored device ID selects reconnect; absence selects QR pairing. whatsmeow documents `Logout()` as unlink + disconnect + local device-store deletion.
- `granola-mcp-plus` requires a logged-in desktop app and has no login/logout implementation of its own.

### Observed locally without inspecting secret values

- Workspace credential discovery resolves to `~/.google_workspace_mcp/credentials`, containing one email-named record and `oauth_states.json`; the directory is mode `0700`.
- WhatsApp state resolves to the two expected SQLite files under the bridge `store/` directory; both are mode `0600`.
- Granola 7.441.4 is installed on macOS. Only encrypted names were found for the two credential files that v1.2.0 expects in plaintext.

### Gaps to carry into existing execution/prototype tickets

- Add sanitized per-VM status/disconnect adapters: Google verified email + API check, WhatsApp logged-in account + `Logout()`, and Granola verified identity. The coordinator must never receive credential material.
- Resolve Granola before the Aviral migration rehearsal: current Linux/macOS path and encrypted-state mismatch is a hard parity blocker, not a state-copy exercise.
- Test Google portal callback routing with a non-production grant and exact registered URI; test WhatsApp state **move** with services stopped and source disabled. Do not test by cloning/running credentials concurrently.
- Use a disposable Granola account/app profile to determine current supported token discovery, identity, sign-out, and revocation behavior. Do not alter Aviral's live Granola state.

These are implementation/validation work already covered by map tickets #62, #64, #68, and #71; no new child question is needed.
